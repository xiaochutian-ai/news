# Dashboard 阶段对比重构设计

## 目标

- 在保持现有三段式模板边界不变的前提下，重构 Dashboard 的三个独立模块：
  - `生成选项`
  - `结果摘要`
  - `生成详情`
- 将 `生成选项` 从普通表单行改为结构化控制面板，按控制维度分区排布：
  - 信息源
  - 过滤策略
  - 时间筛选
  - 去重策略
- 将 `生成详情` 改为高信息密度的阶段对照视图，默认展示：
  - `过滤后`
  - `去重后`
- 保持现有本地 Dashboard 路由、表单提交方式、Jinja 模板体系和后端数据结构可复用，遵守最小化改动原则。

## 背景

- 当前正式 Dashboard 已拆分为三个子模板：
  - `app/templates/_dashboard_generation_options.html.j2`
  - `app/templates/_dashboard_summary.html.j2`
  - `app/templates/_dashboard_stage_details.html.j2`
- 当前预览版 Design A 也使用独立子模板：
  - `app/templates/_preview_a_generation_options.html.j2`
  - `app/templates/_preview_a_summary.html.j2`
  - `app/templates/_preview_a_stage_details.html.j2`
- 正式 Dashboard 当前样式偏轻量，信息密度较低。
- 生成详情当前以“四阶段平铺卡片”展示，缺少阶段间前后对照，不利于快速判断筛选和去重效果。
- 用户已确认最终视觉与交互方向：
  - 基于方案 A 收敛
  - 详情区默认对比 `过滤后 vs 去重后`
  - 左侧阶段条采用“状态优先”
  - 摘要默认折叠
  - 字体减小，提升首屏信息密度

## 需求确认

- `生成选项` 继续保留独立模块，不与 `结果摘要` 或 `生成详情` 混排。
- `生成选项` 中的控制项按语义分仓，不再使用一整行松散堆叠。
- `结果摘要` 继续保留独立模块，但只承载运行决策信息和关键统计，不重复细节列表。
- `生成详情` 继续保留独立模块，但从“阶段平铺”改为“阶段筛选 + 双栏对照”。
- 详情区默认对比 `过滤后 vs 去重后`，因为这是当前最有判断价值的一段。
- 左侧阶段条强调“当前所处阶段”，数量变化作为辅助信息，不抢主视觉层级。
- 摘要内容默认折叠，仅在需要时展开。
- 不引入新前端框架，不引入客户端状态管理，不改后端接口协议。

## 方案对比

### 方案 A：结构化控制台 + 阶段双栏对照

- 做法：
  - 生成选项按控制维度拆成独立分区卡片
  - 生成详情左侧为阶段状态条，右侧为前后阶段双栏对照
  - 默认对比 `过滤后 vs 去重后`
  - 摘要默认折叠
- 优点：
  - 阅读路径清晰
  - 阶段差异一眼可比
  - 结构和现有三段式模板边界天然一致
  - 对当前数据结构要求最低
- 缺点：
  - 需要对详情模板做较大样式和布局重排

### 方案 B：结构化控制台 + 漏斗明细表

- 做法：
  - 生成选项与方案 A 相同
  - 生成详情改为阶段漏斗和表格式通栏追踪
- 优点：
  - 总量变化更直观
  - 更适合阶段数量审计
- 缺点：
  - 单条文章前后去留关系不如双栏对照直观
  - 对阅读密度要求更高，首版更容易拥挤

## 推荐方案

- 采用方案 A：结构化控制台 + 阶段双栏对照。
- 原因：
  - 最符合“每次展示前一个阶段和后一个阶段的对比”这一核心诉求
  - 与当前三模块拆分天然对齐，符合高内聚、低耦合
  - 在不引入新实体的前提下，能明显提高信息密度与可解释性

## 设计

### 模块边界

- `生成选项`
  - 负责本次运行参数输入
  - 只包含可编辑控制项和执行按钮
  - 不包含运行结果或阶段统计
- `结果摘要`
  - 负责本次运行结果总览
  - 只展示关键数量、日期、数据源、策略与产物链接
  - 不承载阶段明细卡片
- `生成详情`
  - 负责文章级明细、阶段切换和前后阶段对比
  - 不承担表单输入职责

### 生成选项布局

- 使用三列结构化排布：
  - 第一列：`信息源`
  - 第二列：`过滤策略` + `去重策略`
  - 第三列：`时间筛选` + 执行按钮
- 保持现有字段名不变：
  - `sources`
  - `filter_strategy`
  - `time_strategy`
  - `dedupe_strategy`
- `信息源` 区以卡片式复选项展示，强化来源辨识。
- `过滤策略` 和 `去重策略` 保持当前后端字段语义，仅调整展示方式和说明文案。
- `时间筛选` 独立成区，不与过滤策略混排。
- 执行按钮固定落在控制区末端，保证操作入口稳定。

### 结果摘要布局

- 保持为独立模块，但收敛为两层信息：
  - 第一层：核心统计
    - 原始候选数
    - 过滤后数量
    - 最终入选数
  - 第二层：运行决策信息
    - 稿件日期
    - 来源日期
    - 数据源
    - 过滤策略
    - 时间策略
    - 去重策略
    - HTML / JSON 产物链接
- 不在结果摘要区重复展示文章列表。
- 统计卡保持高对比，但不要引入与详情区冲突的交互状态。

### 生成详情布局

- 顶层结构改为两栏：
  - 左栏：阶段状态条
  - 右栏：阶段对照面板
- 左栏为固定的四阶段纵向步骤条：
  - 原始候选
  - 过滤后
  - 去重后
  - 最终入选
- 状态条采用“状态优先”：
  - 当前对比阶段高亮更强
  - 非当前阶段弱化
  - 数量作为辅助信息，不抢主层级
- 右栏默认展示两个相邻阶段：
  - 左侧：`过滤后`
  - 右侧：`去重后`
- 用户切换其他阶段时，始终展示“当前阶段的前一阶段 + 当前阶段”：
  - 过滤后：显示 `原始候选 vs 过滤后`
  - 去重后：显示 `过滤后 vs 去重后`
  - 最终入选：显示 `去重后 vs 最终入选`
- `原始候选` 不单独作为当前对比阶段，因为它没有前置阶段，不满足“前后对比”的稳定模式。

### 文章卡片设计

- 卡片字体整体缩小一档，优先保证单屏容纳更多条目。
- 标题、元信息、标签保留常驻显示：
  - 标题
  - 来源
  - 发布时间
  - 分数
  - 标签
- 摘要默认折叠：
  - 使用原生 `details/summary`
  - 默认不展开
  - 仅在用户需要时查看
- 在右侧当前阶段卡片中优先展示：
  - 保留原因
  - 去重原因
  - 差异说明
- 对已被移除的条目，可用弱强调样式显示原因：
  - 与其他文章重复
  - 内容表达高度相似
  - 分数或优先级不足

### 阶段筛选行为

- 不新增前端状态管理框架。
- 使用轻量原生交互实现阶段切换：
  - 正式页可用最小 JavaScript 或纯表单参数回显
  - 预览页保持相同语义
- 需要引入一个轻量的当前阶段变量，例如：
  - `selected_stage`
- 默认值：
  - `deduped_articles`
- 模板根据 `selected_stage` 推导：
  - `previous_stage`
  - `current_stage`
- 阶段对照的数据源全部复用 `result.stages`，不新增后端 trace 实体。

### 数据映射

- 继续复用现有 `result.stages`：
  - `raw_articles`
  - `filtered_articles`
  - `deduped_articles`
  - `chosen_articles`
- 新增一个轻量映射表，仅在模板或页面渲染上下文中使用：
  - `raw_articles -> 无前序`
  - `filtered_articles -> raw_articles`
  - `deduped_articles -> filtered_articles`
  - `chosen_articles -> deduped_articles`
- 不改变 `run_digest()` 返回结构的文章列表字段。
- 如需记录当前阶段选择，只增加页面层渲染参数，不侵入 pipeline。

## 数据流

1. 用户在 `生成选项` 模块选择信息源、过滤策略、时间筛选和去重策略
2. 提交后继续调用现有 `/run` 或预览页运行入口
3. 后端执行 `run_digest()`，返回现有结果结构
4. 页面渲染时增加当前阶段选择参数：
   - 未指定时默认 `deduped_articles`
5. 模板根据当前阶段选择，推导并渲染：
   - 左侧阶段状态条
   - 右侧前一阶段面板
   - 右侧当前阶段面板
6. 文章摘要保持默认折叠，不影响后端数据

## 错误处理

- 当 `result` 为空时：
  - `结果摘要` 展示空态说明
  - `生成详情` 展示“等待执行”空态，不渲染阶段对照
- 当某阶段列表为空时：
  - 当前阶段面板显示空态
  - 仍保留阶段框架，避免布局跳变
- 当阶段选择无效时：
  - 回退到默认值 `deduped_articles`
- 当文章缺少可选字段时：
  - 缺少 `summary` 时不渲染折叠块
  - 缺少 `score` 或 `published_at` 时静默省略对应元信息

## 测试策略

- 模板结构测试：
  - 验证正式页仍由三个独立模块组成
  - 验证 `生成选项` 中出现按控制维度分区的标题文本
  - 验证 `生成详情` 默认展示 `过滤后` 和 `去重后`
- 交互语义测试：
  - 验证默认当前阶段为 `deduped_articles`
  - 验证切换到 `chosen_articles` 时，会展示 `去重后 vs 最终入选`
- 信息密度测试：
  - 验证模板中存在 `details` 折叠结构
  - 验证文章摘要不会在默认状态直接平铺为长文本
- 预览页一致性测试：
  - 验证 preview A 保持与正式页一致的三模块语义
  - 验证 preview A 也采用默认阶段对照和摘要折叠

## 风险与处理

- 风险：正式页和预览页布局分叉过大
  - 处理：先统一结构语义，再允许样式表现有所区分
- 风险：阶段对照逻辑让模板判断变重
  - 处理：把阶段前后关系映射集中在一处，避免模板内散落条件分支
- 风险：信息密度提高后页面显得拥挤
  - 处理：只压缩字号和摘要展开态，不压缩核心留白与卡片边界
- 风险：为对照关系新增过多后端字段
  - 处理：只引入页面层的 `selected_stage`，不新增 trace 模型或数据库字段

## 范围边界

- 本次不改动 `run_digest()` 的主流程语义。
- 本次不新增数据库表或状态持久化字段。
- 本次不引入新前端框架。
- 本次不做跨阶段文章精确 diff 算法。
- 本次不做来源分组二次展开的额外复杂交互，除非复用现有预览版逻辑的成本足够低。
