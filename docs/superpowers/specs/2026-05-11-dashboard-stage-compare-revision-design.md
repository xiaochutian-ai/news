# Dashboard 阶段对照修订设计

## 目标

- 修订 `生成详情` 的展示语义，使其严格满足“前一阶段 vs 当前阶段”的左右对照。
- 修订 `结果摘要` 的布局组织，消除当前卡片高度和信息密度不均的问题。
- 保持现有三模块边界不变：
  - `生成选项`
  - `结果摘要`
  - `生成详情`
- 继续遵守最小化改动原则，不引入新前端框架，不改 `run_digest()` 返回结构。

## 背景

- 当前版本虽然默认只激活一组阶段对照，但模板内部仍预渲染了全部 stage pair，再通过 `hidden` 切换。
- 当前 `生成详情` 的右栏展示的是“当前阶段完整列表”，不是“左栏候选经过条件校验后的结果子集”。
- 当前 `结果摘要` 使用 `3 个统计卡 + 1 个元信息卡` 的布局，导致左侧偏空、右侧偏挤，视觉重心不平衡。
- 用户已确认本轮修订方向：
  - `生成详情` 不做分页
  - 左栏展示前一阶段的全部条目
  - 右栏只展示左栏条目中进入后一阶段的结果
  - 页面一次只展示当前选中的一组阶段对照
  - `结果摘要` 改成更均衡的 `2 x 2` 组织

## 需求确认

- `生成详情` 必须是“当前组对照”，不是“所有组切换面板的集合”。
- 右栏不是后一阶段的全量列表，而是当前左栏集合经过当前阶段筛选后保留下来的子集。
- 默认对照仍为 `过滤后 -> 去重后`。
- 可切换的阶段组只有三组：
  - `原始候选 -> 过滤后`
  - `过滤后 -> 去重后`
  - `去重后 -> 最终入选`
- 不做分页，不增加无限滚动，不增加搜索。
- `结果摘要` 继续只展示摘要信息，不混入文章明细。

## 方案对比

### 方案 A：当前组实时渲染 + 摘要 2 x 2 重排

- 做法：
  - 页面只渲染当前选中的一个 stage pair
  - 左栏展示前一阶段全量条目
  - 右栏通过交集映射展示“左栏中被保留到当前阶段”的结果
  - 摘要区改为四块均衡卡片
- 优点：
  - 最符合最新确认的交互语义
  - DOM 更干净
  - 逻辑更容易向用户解释
  - 摘要区视觉更平衡
- 缺点：
  - 模板需要增加一层当前组数据推导

### 方案 B：保留预渲染全部 stage pair，仅修右栏语义

- 做法：
  - 保留现有 `data-stage-pair` 结构
  - 只把右栏改为“左栏子集”
  - 摘要区同步重排
- 优点：
  - 改动范围略小
- 缺点：
  - 结构语义仍不干净
  - 与“页面一次只展示当前组”的要求不完全一致

## 推荐方案

- 采用方案 A：当前组实时渲染 + 摘要 `2 x 2` 重排。
- 原因：
  - 更贴合最新确认的业务语义
  - 能从结构层面避免“虽然隐藏，但其实全都在页面里”的实现偏差
  - 有利于后续继续做轻量交互，而不积累模板复杂度

## 设计

### 模块边界

- `生成选项`
  - 不变
  - 继续负责输入参数控制
- `结果摘要`
  - 只负责展示结构化运行摘要
  - 不展示文章卡片
- `生成详情`
  - 只负责当前选中阶段组的左右对照
  - 不再承载全部阶段组的预渲染面板

### 生成详情布局

- 顶部保留阶段切换条，但它只用于切换“当前组”：
  - `原始候选 -> 过滤后`
  - `过滤后 -> 去重后`
  - `去重后 -> 最终入选`
- 主体固定为左右两栏：
  - 左栏：前一阶段全部条目
  - 右栏：左栏条目中通过当前阶段条件后保留的条目
- 默认阶段组：
  - 左栏：`过滤后`
  - 右栏：`去重后`
- 页面一次只渲染当前选中的一个 stage pair。

### 生成详情数据语义

- 左栏数据：
  - 直接取当前组 `previous_key` 对应的全部文章列表
- 右栏数据：
  - 不是直接渲染 `current_key` 全量列表
  - 而是从左栏文章中筛出“也存在于 `current_key` 列表中的条目”
- 交集判断规则：
  - 优先使用 `url`
  - 如 `url` 不可用，则回退到 `title + source`
- 由此保证右栏语义始终是：
  - “左栏这些候选里，哪些进入了下一阶段”

### 生成详情文案与统计

- 左栏头部展示：
  - 阶段名称
  - 总条数
  - 阶段说明
- 右栏头部展示：
  - 阶段名称
  - 通过条数
  - 淘汰条数
- 右栏统计计算：
  - 通过数 = 右栏条目数
  - 淘汰数 = 左栏条目数 - 右栏条目数
- 当右栏为空时：
  - 展示空态说明
  - 文案建议：`当前组中没有条目进入下一阶段`

### 文章卡片

- 左右两栏卡片结构保持一致：
  - 标题
  - 来源
  - 时间
  - 分数
  - 标签
  - 折叠摘要
- 右栏卡片额外保留一条状态说明：
  - 过滤阶段：`通过主题过滤`
  - 去重阶段：`通过去重校验`
  - 最终入选阶段：`进入最终晨报草稿`
- 不再额外渲染“占位的移除卡片”。
- 不做分页，不做分组二次展开。

### 结果摘要布局

- 从当前的 `3 + 1` 布局改为均衡的 `2 x 2`：
  - 左上：`核心统计`
  - 右上：`阶段转化`
  - 左下：`运行配置`
  - 右下：`产物入口`
- `核心统计`
  - 展示：
    - 原始候选数
    - 过滤后数量
    - 最终入选数
- `阶段转化`
  - 展示关键转化关系：
    - `原始 -> 过滤后`
    - `过滤后 -> 最终入选`
  - 可以补充简洁比例或转化率，但不强制
- `运行配置`
  - 展示：
    - 稿件日期
    - 来源日期
    - 数据源
    - 过滤策略
    - 时间策略
    - 去重策略
- `产物入口`
  - 单独承载：
    - HTML
    - JSON
- 数据源展示方式：
  - 使用紧凑 badge 或多行流式排列
  - 避免一整行长文本造成卡片拥挤

### 交互实现

- 继续使用轻量前端交互。
- 允许保留最小 JavaScript 进行阶段切换。
- 但切换逻辑改为：
  - 根据当前选中阶段组，更新单一对照容器中的左右栏内容
  - 或由模板直接渲染当前组，不保留全部组 DOM
- 两种实现中，优先选择更小改动且可维护的方式。

### 数据映射

- 继续复用 `result.stages`：
  - `raw_articles`
  - `filtered_articles`
  - `deduped_articles`
  - `chosen_articles`
- 新增轻量映射只存在于页面层：
  - 当前阶段组 `selected_stage`
  - 当前组的 `previous_items`
  - 当前组的 `current_items`
  - 当前组的 `passed_items`
- 不改 pipeline，不改存储层，不改数据库。

## 数据流

1. 用户执行晨报生成
2. 后端返回现有 `result.stages`
3. 页面确定当前选中阶段组，默认 `deduped_articles`
4. 根据阶段组映射得到：
   - `previous_items`
   - `current_items`
5. 使用交集规则，从 `previous_items` 中筛出 `passed_items`
6. 左栏渲染 `previous_items`
7. 右栏渲染 `passed_items`
8. 摘要区按 `2 x 2` 模式渲染四块信息卡

## 错误处理

- 当 `result` 为空时：
  - `结果摘要` 显示等待执行空态
  - `生成详情` 显示等待执行空态
- 当当前组左栏为空时：
  - 左栏显示空态
  - 右栏同步显示空态
- 当右栏为空但左栏有数据时：
  - 左栏正常显示
  - 右栏显示“当前组中没有条目进入下一阶段”
- 当阶段选择无效时：
  - 回退到默认 `deduped_articles`

## 测试策略

- 结构测试：
  - 验证 `生成详情` 同时只渲染一组左右对照
  - 验证默认对照为 `过滤后 -> 去重后`
- 语义测试：
  - 验证右栏内容是左栏与当前阶段列表的交集，而不是当前阶段全量
  - 验证切换到 `chosen_articles` 时，左右栏变为 `去重后 -> 最终入选`
- 摘要测试：
  - 验证摘要区存在 `核心统计 / 阶段转化 / 运行配置 / 产物入口`
  - 验证数据源以紧凑列表或 badge 渲染，而不是单行长文本堆叠
- 折叠测试：
  - 验证摘要仍使用 `details` 折叠，不回退为长文本平铺

## 风险与处理

- 风险：交集判断不稳定
  - 处理：优先使用 `url`，回退 `title + source`
- 风险：模板内计算过重
  - 处理：把阶段映射和交集计算集中在一处，避免散落判断
- 风险：摘要重排后 preview 与正式页差异过大
  - 处理：统一信息结构，允许视觉样式不同

## 范围边界

- 本次不新增分页。
- 本次不新增搜索、排序或筛选器。
- 本次不做跨阶段精确 diff 标注。
- 本次不修改 `run_digest()` 输出协议。
